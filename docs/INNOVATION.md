### 1. 跨压缩包搜索引擎 (Cross-Archive Search)

#### 概念

像 Everything 搜索硬盘一样，**跨所有压缩包搜索内部文件名甚至文件内容**。这是目前没有任何主流压缩软件能做到的功能。

#### 用户场景

- "我记得 2024 年某个压缩包里有一张叫 DSC_0234.jpg 的照片，在哪个包里？"
- "我要找所有压缩包里的 README.md 文件"
- "搜索所有压缩包中，包含 password 字样的配置文件"
- "哪个压缩包最大？哪个压缩包最近没被测试过？"

#### 核心设计

`
┌─────────────────────────────────────────────────┐
│ Cross-Archive Search                      🔍 ✕  │
├─────────────────────────────────────────────────┤
│ [所有目录]  [扫描位置: D:\Archives\]             │
│ ┌─────────────────────────────────────────────┐ │
│ │ 搜索结果: 47 项  (索引: 328 个压缩包)        │ │
│ │                                             │ │
│ │ 📦 工作备份_2024.7z                         │ │
│ │   └── 📄 DSC_0234.jpg         2.3 MB        │ │
│ │   └── 📄 DSC_0235.jpg         1.8 MB        │ │
│ │ 📦 照片合集.zip                             │ │
│ │   └── 📄 DSC_0234.jpg         3.1 MB        │ │
│ │ 📦 源码归档.7z                              │ │
│ │   └── 📄 README.md           12 KB  ← 含 pwd │ │
│ │                                             │ │
│ │ [双击结果] → [打开压缩包并定位到该文件]       │ │
│ └─────────────────────────────────────────────┘ │
│ 索引状态: 328 个包 / 12,345 项 / 上次更新: 刚刚  │
│ [立即扫描] [增量更新] [排除规则]                  │
└─────────────────────────────────────────────────┘
`

#### 技术要点

- **索引层**: 本地 SQLite 数据库存储所有压缩包的条目元数据（文件名、路径、大小、CRC32、所属压缩包路径）
- **实时搜索**: 如果索引命中直接返回，未索引的压缩包实时 ListEntriesAsync 查询
- **全文搜索**: 可选对 .txt/.cs/.md/.xml 等文本文件提取内容，用 SQLite FTS 或 Lucene 索引
- **增量更新**: 监听文件系统变更（FileSystemWatcher），压缩包变化时自动更新索引
- **排除规则**: 可配置跳过加密包、超大包（>1GB）、特定目录
- **从搜索结果打开**: 双击结果 → 调用 --open 并定位到对应条目

#### 与现有系统的集成

- 复用所有引擎的 ListEntriesAsync
- 复用密码管理器的自动尝试逻辑处理加密包
- 搜索结果 UI 可参考已有文件列表的虚拟化展示

#### 工作量评估

| 模块 | 工时 | 说明 |
|------|:----:|------|
| Core: 索引引擎（扫描 + SQLite 存储 + 搜索） | 6-8h | 核心逻辑 |
| Core: 增量更新（FileSystemWatcher + 变更检测） | 2-3h | 被动索引同步 |
| UI: 搜索窗口（结果列表 + 过滤 + 排序） | 3-4h | 新窗口 |
| UI: 从搜索结果打开压缩包 | 1h | 适配已有 --open |
| 可选: 全文索引 | +4-6h | 文本提取+FTS |

**总计**: ~12h（基础版）/ ~18h（含全文搜索）

### 2. 压缩健康仪表盘 (Archive Health Dashboard)

#### 概念

全盘压缩包资产健康看板 — 定期扫描完整性，展示所有压缩包的"健康状况"。让你从一个仪表盘知道自己所有压缩包是否完好。

#### 用户场景

- "我 3 年前备份的压缩包还能打开吗？"
- "有没有压缩包开始出现坏块了（位衰减）？"
- "哪些压缩包最大/最小？哪些占的空间最多？"
- "哪些压缩包我从来没测试过完整性？"

#### 核心设计

`
┌─────────────────────────────────────────────────────────┐
│ 压缩健康仪表盘                            🛡️ [立即扫描] │
├─────────────────────────────────────────────────────────┤
│ 总体概览                                                 │
│ ✅ 正常: 285 个 (86%)                                     │
│ ⚠️ 警告:  32 个 (10%)  ── 上次测好这次测坏                │
│ ❌ 损坏:  12 个  (4%)  ── 无法列出内容                    │
│ 🔒 加密:   8 个  (2%)  ── 需密码未测试                    │
│                                                           │
│ 📊 存储使用: 总共 85.2 GB / 已用 62.1 GB                   │
│ 📈 平均压缩比: ZIP 62% · 7z 48% · RAR 55%                 │
│                                                           │
│ ┌─────────────────────────────────────────────────────┐   │
│ │ 最近损坏 / 警告的压缩包                               │   │
│ │ ⚠️ 工作备份_2024.7z    24.3 MB  上次: ✅ 2026-05-01  │   │
│ │ ❌ 照片合集.zip         1.2 GB   损坏: 3个条目无法读取 │   │
│ │ ⚠️ 项目源码.tar.gz     45.0 MB  CRC 不匹配            │   │
│ │ [查看全部]                                            │   │
│ └─────────────────────────────────────────────────────┘   │
│                                                           │
│ ⏰ 下次全量扫描: 2026-06-15 03:00 (每周)                   │
│ ☐ 损坏时发送通知                                          │
└─────────────────────────────────────────────────────────┘
`

#### 技术要点

- **扫描核心**: 复用 TestArchiveAsync（v0.3.10 已升级为完整完整性验证）
- **数据库**: SQLite 存储每次扫描结果、时间戳、错误信息，支持趋势分析
- **警告检测**: 压缩包上次测试完好、这次测试失败 → 标记为 ⚠️（可能是位衰减）
- **后台计划任务**: 内置定时器或 Windows 计划任务，低优先级后台扫描
- **通知**: 检测到损坏时弹出 Windows Toast 通知
- **过滤与排序**: 按健康状态/大小/日期/路径排序，支持搜索

#### 与现有系统的集成

- 完整复用 IArchiveEngine.TestArchiveAsync
- 引用已有密码管理器处理加密包
- 扫描结果存储可与提取日志（extract-journal-undo 计划）共用 SQLite 数据库

#### 工作量评估

| 模块 | 工时 | 说明 |
|------|:----:|------|
| Core: 扫描引擎 + SQLite 持久化 | 4-6h | 核心 |
| Core: 定时任务调度 | 2-3h | 计划系统 |
| UI: 仪表盘窗口 | 4-6h | 图表 + 列表 |
| UI: 损坏通知（Toast） | 1h | 系统通知 |

**总计**: ~11-16h

### 3. 压缩包标签与知识库系统 (Archive Knowledge Base)

#### 概念

把 MantisZip 从一个压缩工具变成**个人归档管理器**。给每个压缩包打标签、写描述、评星级，像音乐播放器管理曲库一样管理你的归档文件。

#### 用户场景

- "帮我找出所有标了'税务'标签的 2023 年压缩包"
- "这个压缩包是干什么的来着？" → 打开看到自己写的描述
- 批量管理：给所有含"照片"的压缩包自动加"图片"标签
- 按标签/描述/日期/大小组合搜索过滤

#### 核心设计

`
┌─────────────────────────────────────────────────┐
│ 归档知识库                          [+ 新建标签] │
├─────────────────────────────────────────────────┤
│ 标签过滤: [🏷 全部] [📷 照片] [💼 工作] [🏠 个人] │
│                                                       │
│ ┌─────────────────────────────────────────────────┐   │
│ │ ☐ 📦 2024-北京旅行.zip         📷 照片  🏠 个人   │   │
│ │ │  "2024 年 10 月北京故宫/长城照片"  ★★★★☆       │   │
│ │ │  345 MB · 127 个文件 · 2024-10-15              │   │
│ │ ├─────────────────────────────────────────────   │   │
│ │ ☐ 📦 项目源码_v3.1.zip        💼 工作             │   │
│ │ │  "v3.1 正式版发布源码，含文档更新"   ★★★★★       │   │
│ │ │  12 MB · 856 个文件 · 2026-05-20              │   │
│ │ └─────────────────────────────────────────────   │   │
│ └─────────────────────────────────────────────────┘   │
│                                                       │
│ 📊 共 128 个压缩包 · 已标记 93 个 (73%)                │
│ [批量标记] [自动标签规则] [导出目录]                    │
└─────────────────────────────────────────────────┘
`

#### 技术要点

- **存储**: 单独的 SQLite 库，存储标签表、压缩包-标签关联表、压缩包元数据缓存
- **自动标签规则**: 基于文件名模式/路径/大小/格式自动分配标签（如 *.jpg → "图片"）
- **星级评分**: 1-5 星，方便快速筛选重要/有价值的压缩包
- **全文搜索**: 标签 + 描述 + 压缩包名联合搜索
- **批量操作**: 多选压缩包 → 批量加/删标签、批量写描述

#### 与现有系统的集成

- 与跨压缩包搜索引擎共享 SQLite 数据库（可合并为一个归档数据库）
- 标签可显示在已有文件列表的额外列
- 右键菜单可添加"编辑标签"项

#### 工作量评估

| 模块 | 工时 | 说明 |
|------|:----:|------|
| Core: 知识库数据模型 + SQLite | 3-4h | 数据层 |
| Core: 自动标签规则引擎 | 2-3h | 规则系统 |
| UI: 知识库主窗口 | 4-5h | 列表 + 筛选 + 编辑 |
| UI: 标签编辑对话框 | 1-2h | 编辑单个 |
| UI: 右键菜单集成 | 0.5h | 快捷操作 |

**总计**: ~10-14h

### 4. 智能格式建议 (Smart Format Advisor)

#### 概念

选中文件后自动分析内容，推荐最优压缩策略并**解释原因**。降低非专业用户的决策成本。

#### 用户场景

- "我要压缩一堆照片，用 ZIP 还是 7z？" → 软件告诉你"JPG 已经压缩过了，ZIP 就够了"
- "这些文件哪种格式能最小？" → 软件分析后给出预估和推荐
- "为什么你推荐 7z 而不是 ZIP？" → 显示详细理由

#### 核心设计

`
┌─────────────────────────────────────────────────┐
│ 智能压缩建议                                      │
├─────────────────────────────────────────────────┤
│ 📊 内容分析结果:                                   │
│                                                   │
│ 35 个文件 · 总计 156 MB                           │
│                                                   │
│ 🖼 图片 (20 .jpg)        98 MB  ─── 已高度压缩    │
│ 📝 文本 (10 .txt)        12 MB  ─── 非常适合压缩  │
│ 📦 二进制 (5 .exe)       46 MB  ─── 中等压缩潜力  │
│                                                   │
│ ┌─────── 推荐方案 ──────────────────────────────┐ │
│ │ 🥇 7z + 极限压缩                              │ │
│ │   预计大小: 42 MB (节省 73%)                   │ │
│ │   理由: 文本占比较大，7z 的 LZMA2 可充分压缩   │ │
│ │                                                │ │
│ │ 🥈 ZIP + 标准压缩                              │ │
│ │   预计大小: 68 MB (节省 56%)                   │ │
│ │   理由: 兼容性最好，无需额外软件解压            │ │
│ │                                                │ │
│ │ 🥉 7z + 快速压缩                               │ │
│ │   预计大小: 50 MB (节省 68%)                    │ │
│ │   理由: 平衡速度与体积                          │ │
│ └─────────────────────────────────────────────┘ │
│                                                   │
│ [使用推荐方案压缩]    [自定义]                     │
└─────────────────────────────────────────────────┘
`

#### 技术要点

- **文件类型分析**: 基于扩展名 + 魔数检测（已有 preview-magic-detection 计划基础）
- **压缩潜力估算**: 每种文件类型预设压缩率系数，乘以文件大小得出预估值
  - .jpg/.png/.mp4: 95-98%（几乎不能再压缩）
  - .txt/.docx: 20-30%（文本类压缩率高）
  - .exe/.dll: 40-55%（二进制可压缩）
- **格式推荐逻辑**: 决策树 — 是否需要跨平台兼容 → ZIP；追求最小体积 → 7z；需要分卷 → 7z
- **压缩等级建议**: 文本多 → 极限；已压缩格式多 → 标准/快速
- **用户反馈循环**: 用户实际压缩后记录真实压缩率，用于校准预估模型

#### 与现有系统的集成

- 可嵌入 CompressSettingsWindow 作为默认推荐面板
- 复用魔数检测结果（计划中）进行精确的文件类型判断
- 压缩历史记录可用于校准压缩率模型

#### 工作量评估

| 模块 | 工时 | 说明 |
|------|:----:|------|
| Core: 文件分析 + 压缩率模型 | 4-6h | 核心算法 |
| Core: 决策推荐引擎 | 2-3h | 推荐逻辑 |
| UI: 建议面板 | 2-3h | 嵌入压缩窗口 |
| 可选: 学习校准 | +2-3h | 记录+更新模型 |

**总计**: ~8-12h（基础版）/ ~12-15h（含学习校准）

### 5. 提取前冲突全景地图 (Pre-extract Conflict Preview)

#### 概念

在真正提取文件之前，完整展示目标路径下的冲突情况，让用户在一个视图中做所有决策。而不是提取到一半弹单个对话框。

#### 用户场景

- 提取一个大型压缩包前想看看哪些文件会覆盖现有文件
- 想对比目标路径已存在的文件与压缩包内的版本差异
- 对不同冲突文件应用不同策略（覆盖旧的、跳过大的、重命名等）

#### 核心设计

`
┌─────────────────────────────────────────────────────────┐
│ 解压冲突预览                  目标: D:\Documents\       │
├─────────────────────────────────────────────────────────┤
│ 共 45 个文件 · 12 个冲突                                 │
│                                                           │
│ ┌─────────────────────────────────────────────────────┐   │
│ │ ☐ 状态 │ 文件名          │ 包内        │ 目标       │   │
│ │ ├───────┼─────────────────┼─────────────┼───────────┤   │
│ │ │ ⚠️  │ report.docx      │ 2026-06-01  │ 2026-05-15│   │
│ │ │ 🔴  │ photo.jpg        │ 3 MB        │ 5 MB      │   │
│ │ │ 🟢  │ notes.txt        │ 12 KB       │ 12 KB     │   │
│ │ │ ➕  │ new_file.cs      │ ─           │ ─         │   │
│ │ └─────┴─────────────────┴─────────────┴───────────┘   │
│ │                                                   │   │
│ │ 选中项操作: [覆盖] [跳过] [重命名] [对比内容]       │   │
│ │ ☑ 应用到全部同类冲突                               │   │
│ └─────────────────────────────────────────────────────┘   │
│                                                           │
│ 批量策略:                                                   │
│ ◉ 所有冲突一律 [覆盖]                                      │
│ ○ 覆盖旧的 / 覆盖小的                                      │
│ ○ 全部跳过（只提取不冲突文件）                              │
│ ○ 全部自动重命名                                           │
│                                                           │
│ [开始提取]  [取消]                                         │
└─────────────────────────────────────────────────────────┘
`

#### 技术要点

- **冲突检测**: 在提取前扫描目标路径，与压缩包文件列表做交叉对比
- **对比维度**: 文件名、大小、修改时间，可选深度对比（SHA256）
- **状态指示**: 🟢 完全一致 / 🟡 同名但属性不同 / 🔴 严重冲突 / ➕ 无冲突
- **批量策略**: 快速应用全局规则到所有冲突项
- **对比内容快捷操作**: 选中两个文件 → 显示大小/时间差异，未来可调用 ArchiveDiff

#### 与现有系统的集成

- 复用冲突解决已有的 CompressConflictDialog 逻辑
- 对话框出现时机从提取中提前到提取前
- 可与提取配置窗口（ExtractSettingsWindow）集成

#### 工作量评估

| 模块 | 工时 | 说明 |
|------|:----:|------|
| Core: 冲突检测引擎 | 3-4h | 扫描+对比 |
| UI: 冲突全景窗口 | 4-6h | 列表+批量策略 |
| UI: 提取流程重构 | 2-3h | 提前触发点 |

**总计**: ~9-13h

### 6. 文件夹监控自动归档 (Folder Watch Auto-Archive)

#### 概念

监控指定文件夹，新文件出现时按规则自动压缩归档。类似 Hazel（macOS）/ DropIt 但内置于压缩软件。

#### 用户场景

- "Downloads 文件夹里的 .exe/.msi 安装包，超过 7 天自动压缩归档"
- "桌面截图自动压缩到 Screenshots.zip"
- "日志文件超过 100MB 自动压缩并删除源文件"
- "每天凌晨 3 点压缩 Documents 中的增量文件"

#### 核心设计

`
┌─────────────────────────────────────────────────┐

### 7. 文件边界智能分卷 (Smart Split at File Boundaries)

#### 概念

传统分卷按固定字节大小切分（100MB/卷），可能导致一个文件被劈成跨卷的两半。智能分卷保证每个卷包含完整文件，每个卷独立可用。

#### 用户场景

- 分卷压缩大文件到 U 盘，希望每卷单独解压就能用
- 上传大压缩包到有限制单文件大小的云盘（如 100MB 限制）
- 希望第一卷包含最常用的文件，方便紧急情况快速获取

#### 核心设计

`
传统分卷（7-Zip 行为）:
  backup.7z.001 = 100 MB — 把 bigfile.bin 劈成两半
  backup.7z.002 = 100 MB — 包含 bigfile.bin 后半 + 其他文件
  backup.7z.003 = 45 MB  — 剩余文件
  → 单独解压 .001 什么都用不了

智能分卷（MantisZip 行为）:
  backup.part1.7z = 98 MB — file_a.zip + file_b.zip + file_c.iso（全部完整）
  backup.part2.7z = 95 MB — bigfile.bin（完整）
  backup.part3.7z = 42 MB — 剩余文件（完整）
  → 单独解压 .part1 就能拿到前三个完整文件
`

#### 技术要点

- **预计算**: 先获取所有文件列表和大小，运行背包算法计算最佳文件组合
- **分卷大小**: 用户指定单卷大小上限，算法尽量填满但不超过上限
- **命名方案**: {name}.part{N}.{ext} 替代 {name}.{ext}.{N:03d}
- **可选优先级**: 用户可以设置哪些文件优先放到前面的卷
- **格式支持**: 仅 7z（原生分卷支持），ZIP 不支持文件级分卷

#### 与现有系统的集成

- 在 CompressSettingsWindow 分卷选项旁增加"智能分卷"开关
- 复用 SharpSevenZipCompressor 的卷容量设定（VolumeSize）

#### 工作量评估

| 模块 | 工时 | 说明 |
|------|:----:|------|
| Core: 文件边界算法（背包算法） | 3-5h | 核心逻辑 |
| UI: 配置界面（优先级+开关） | 1-2h | 嵌入已有 |
| Core: 分卷写入适配 | 1-2h | SharpSevenZip 集成 |

**总计**: ~5-9h

### 8. 资源占用控制台 (Resource Governor)

#### 概念

精确控制压缩操作对系统资源的占用，让压缩在后台运行时不影响正常工作。比 WinRAR 的"后台/低优先级"两档粗粒度控制精细得多。

#### 核心设计

`
┌─────────────────────────────────────────────────┐
│ 资源占用控制                        [恢复默认]   │
├─────────────────────────────────────────────────┤
│                                                   │
│ CPU 核心使用:                                      │
│ [═══════────]  4 / 8 核                           │
│                                                   │
│ 内存上限:                                          │
│ [════════──]  1024 MB                             │
│                                                   │
│ 磁盘 IO 优先级:                                    │
│ ○ 不限制                                          │
│ ◉ 低优先级（不影响其他程序）                        │
│ ○ 仅空闲时运行                                    │
│                                                   │
│ 智能节流:                                          │
│ ☑ 电池供电时自动降速                               │
│ ☑ 检测到用户活跃时降低                             │
│ ☑ 前台全屏程序（游戏/视频）时暂停                  │
│                                                   │
│ 当前状态:                                          │
│ 🟢 运行中 · 使用 35% CPU · 420 MB · 低 IO         │
│       · 用户活跃中 → 已自动降速                     │
└─────────────────────────────────────────────────┘
`

#### 技术要点

- **CPU 控制**: 任务并行度（ParallelOptions.MaxDegreeOfParallelism）限制线程数
- **内存控制**: 限制引擎内部缓冲区大小（如 ZipEngine 的流式处理缓冲区）
- **IO 优先级**: ProcessPriorityClass.Idle + 自行控制写入频率
- **用户活跃检测**: 最后一次键盘/鼠标输入时间（GetLastInputInfo API）
- **全屏检测**: 前台窗口是否为全屏（DirectX 游戏/全屏视频）
- **电池感知**: SystemInformation.PowerStatus 检测电池状态

#### 与现有系统的集成

- 设置窗口新增"资源控制"标签页
- 压缩/解压引擎读取 ResourceGovernor 设置的并行度参数
- 进度窗口可显示当前资源占用状态

#### 工作量评估

| 模块 | 工时 | 说明 |
|------|:----:|------|
| Core: 资源控制参数模型 | 1-2h | 配置数据结构 |
| Core: 用户活跃/全屏检测 | 2-3h | Win32 API |
| Core: 引擎适配（并行度/缓冲区） | 3-4h | 各处集成 |
| UI: 控制面板 | 2-3h | 滑动条+选项 |

**总计**: ~8-12h

### 15. 编码检测与文件名乱码修复 (Encoding Detective)

#### 概念

检测压缩包内文件名的真实编码，预览不同编码下的文件名显示，一键修复乱码。

#### 用户场景

- "从日本朋友收到的 ZIP，文件名全是乱码，怎么办？"
- "这个压缩包是 CP932 编码的，但系统用 CP936，显示一堆 _____"
- "批量修复一批遗留压缩包的文件名编码"

#### 核心设计

#### 技术要点

- **编码检测**: 对字节序列用多种编码解码 → 计算可读性评分（有效字符比例、Unicode 块分布）
- **检测候选列表**: CP932/CP936/CP437/CP950/CP737/ISO-8859-1/UTF-8
- **修复方式**: 重建 ZIP 目录（Central Directory）用正确编码写回
- **回滚支持**: 修改前备份原 Central Directory 到临时文件
- **批量操作**: 选中多个压缩包 → 批量检测+修复

#### 与现有系统的集成

- MantisZip 已有 GBK 编码处理（ZipEngine 的 StringCodec），可在此基础扩展
- 编码检测 UI 可以集成到 MainWindow 的"工具"菜单
- 可配合已有的 ZipCommentHelper 实现原地 ZIP 修改
